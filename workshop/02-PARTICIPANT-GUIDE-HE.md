# מדריך למשתתף — חמש לולאות עבודה

היום לא מנסים להספיק מוצר שלם. בונים חתך אחד שאפשר להסביר, לבדוק ולהראות בראיון.

## לפני שמתחילים

```bash
claude --version
npm install
npm run verify
npm run dev
```

פתחו את `http://localhost:4310`. ב-Claude Code בדקו `/status`, `/model` ו-`/permissions`. תנו רק את ההרשאות שנחוצות לתרגיל. אל תכניסו secrets ואל תחברו סביבת production.

כל התוצרים נשמרים ב-`workshop-output/`:

- `PRODUCT-MAP.md` — איך המוצר מחובר.
- `GOAL.md` — היעד, הגבולות והראיות.
- `FEATURE-SPEC.md` — החלטת מוצר אחת מכל הזוויות.
- `PLAN.md` — קבצים, contracts, סיכונים ובדיקות.
- `DECISIONS.md` — החלטות ופשרות שעשיתם.
- `EVIDENCE/` — screenshots, תוצאות tests והשוואות.

## לולאה 1 — להבין ולנעול יעד · 20 דקות

בקשו מ-Claude Code לקרוא את ה-repo וליצור רק את `PRODUCT-MAP.md` ואת `GOAL.md`. בשלב הזה לא מציעים UI ולא כותבים קוד.

`GOAL.md` צריך לענות על ארבע שאלות: מה ישתנה עבור המשתמש, מה לא נכנס לפיצ׳ר, איך נדע שהוא עובד ומתי עוצרים ושואלים בן אדם.

סיימתם כשאדם אחר יכול לקרוא את שני הקבצים ולהצביע על גבול ברור ועל ראיה מדידה.

## לולאה 2 — לכתוב Spec עם מומחים · 30 דקות

הפעילו את `/write-feature-spec` על `workshop/01-PROJECT-BRIEF-HE.md`. בקשו מהסוכן הראשי להפעיל ארבעה מומחים: Product, Design, Architecture ו-QA.

כל מומחה צריך להחזיר שתי החלטות, סיכון אחד ושאלה אחת שעדיין פתוחה. הסוכן הראשי מחבר את התשובות ל-`FEATURE-SPEC.md`; אתם מכריעים במחלוקות. מומחים מייעצים — הם לא מנהלים את המוצר.

סיימתם כשיש החלטת מוצר אחת, non-goals, states, contracts, מצבי כשל וקריטריוני קבלה שאפשר לבדוק.

## לולאה 3 — לארוז מומחיות ב-Skill · 25 דקות

פתחו `http://localhost:4310/design-demo` והריצו בקשה אחת בלי Skill. שמרו את התוצאה ב-`workshop-output/EVIDENCE/before.md`.

העתיקו את `workshop/templates/my-skill/` אל `.claude/skills/my-skill/` וכתבו `SKILL.md` קצר עם מועד הפעלה, שאלות חובה, כללים מקצועיים, פורמט פלט ורובריקה של ארבעה ממדים. הגדירו לכל ממד מה פירוש 5 ומה פירוש 10, ואז קבעו רף מעבר.

תנו ל-baseline ציון מול הרובריקה. הריצו שוב את אותה בקשה עם ה-Skill, בקשו critique על שלושת הפערים בעלי ההשפעה הגבוהה ביותר, תקנו ותנו ל-reviewer טרי לציינן שוב. שמרו `workshop-output/EVIDENCE/after.md`, `workshop-output/EVIDENCE/before-after.md` ואת הציונים.

סיימתם כשעברתם את הרף שהגדרתם, או כשהגעתם ל-blocker, לתקציב או לשלושה סבבים. צריך להיות אפשר להצביע על שלוש החלטות שהשתפרו — לא רק על צבעים או ניסוח שונים.

## לולאה 4 — לתכנן לעומק ולבצע בחתכים · 40 דקות

היכנסו ל-Plan Mode והפעילו `/architecture-plan`. דרשו רשימת קבצים מדויקת, זרימת נתונים ו-contracts, מצבי כשל, בדיקות לכל acceptance criterion וחתך אנכי ראשון שאפשר להשלים בזמן.

שמרו את התוכנית ב-`PLAN.md`, עברו עליה ואשרו אותה. רק אז עברו לביצוע.

בביצוע, עבדו ב-chunks תחומים. אחרי כל chunk הריצו בדיקה צרה ועדכנו את `DECISIONS.md`. אם התגלתה הנחה שמשנה את ה-Spec, עצרו וחזרו לתכנון. אל תתנו לסוכן להמציא את הדרישה תוך כדי קידוד.

סיימתם כשנתיב משתמש אחד עובד מקצה לקצה וה-diff נשאר בתוך גבולות התוכנית.

## לולאה 5 — ביקורת עצמאית וראיות · 15 דקות

הפעילו את `change-reviewer` עם context מבודד. בקשו ביקורת מול `GOAL.md`, `FEATURE-SPEC.md` וה-diff — לא סיכום כללי של הקוד.

1. הריצו בדיקה צרה.
2. הריצו `npm run verify`.
3. תקנו blockers בלבד.
4. הריצו review נוסף אם התיקון שינה התנהגות.
5. עצרו אחרי עד שלושה סבבים.

שמרו ב-`workshop-output/EVIDENCE/` ראיה לכל criterion. אם אי אפשר להוכיח criterion, הוא עדיין לא גמור.

## Capstone — איך מציגים את זה בראיון

הכינו דמו של שלוש דקות: הבעיה והמשתמש, ההחלטות שקיבלתם לפני קוד, מה האצלתם, איך ה-Skill שינה את התוצאה, איפה עצרתם את הלולאה ואיזו ראיה מוכיחה שהחתך עובד.

הסיפור הוא לא "Claude בנה לי". הסיפור הוא איך ניהלתם מערכת עבודה שהגיעה לתוצאה שאפשר להגן עליה.
