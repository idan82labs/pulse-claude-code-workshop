# מדריך למשתתף

## לפני שמתחילים

בדקו ש-Claude Code מותקן ושאתם מחוברים לחשבון. אחר כך הריצו:

```bash
claude --version
npm install
npm run verify
npm run dev
```

פתחו את `http://localhost:4310` והכירו את המוצר. ב-Claude Code בדקו `/status`, `/model` ו-`/permissions`. תנו רק את ההרשאות שנחוצות לתרגיל; אל תכניסו secrets ואל תחברו סביבת production.

אם ההתקנה או ההתחברות נתקעות, הצטרפו למשתתף עם סביבת עבודה תקינה והמשיכו יחד מה-repo. לא משתפים חשבון, סיסמה או token.

## שלב 0 — מיפוי וניסוי effort

לפני ה-Spec, מפו את המוצר: אילו מסכים קיימים, איך ה-API וה-domain מתחברים, איפה הבדיקות ואיך הנתונים זורמים. שמרו את המפה ב-`workshop-output/PRODUCT-MAP.md`.

עכשיו הריצו בדיוק את אותו prompt של plan: פעם אחת עם `/effort low` ופעם אחת עם `/effort high`. השוו את ההנחות, הקבצים, ה-contracts, מצבי הכשל והבדיקות. שמרו את המסקנה ב-`workshop-output/EFFORT-COMPARISON.md`, ואז החזירו `/effort auto`. בניסוי הזה לא כותבים קוד.

סיימתם כשיש `PRODUCT-MAP.md` ו-`EFFORT-COMPARISON.md`, וברור לכם איזו רמת effort מספיקה למשימה.

## שלב 0.5 — מה נכנס לאן

לפני שאתם מוסיפים מנגנון, החליטו מה באמת צריך:

- כלל repo קבוע וקצר → `CLAUDE.md`.
- ידע או workflow שמפעילים לפי צורך → Skill.
- מחקר רועש או מומחה שצריך context מבודד → subagent.
- נתון או פעולה במערכת חיצונית → MCP בהרשאה צרה.
- בדיקה או חסימה שחייבות לרוץ באירוע קבוע → hook.
- חבילה שמפיצה כמה מהיכולות האלה לצוות → plugin.

תרגיל קצר: סווגו lint אחרי edit, ticket מ-Jira, security reviewer, naming rules, design rubric וחבילת צוות. ליד כל אחד כתבו גם מה אסור לו לעשות.

השלב הזה סגור כשבחרתם את המנגנון הפשוט ביותר לכל צורך והגדרתם לפחות גבול הרשאה אחד.

## שלב 1 — מחקר ואפיון

הפעילו את `/write-feature-spec` על הבריף שב-`workshop/01-PROJECT-BRIEF-HE.md`.

בקשו מהסוכן הראשי להתייעץ עם ארבעת המומחים, אחד בכל פעם, ואז לחבר את התובנות למסמך אחד. המומחים מציעים; אתם מחליטים.

בסוף צריך להיות `workshop-output/FEATURE-SPEC.md` עם יעד, non-goals, קריטריונים ותוכנית אימות.

## שלב 2 — ידע מקצועי שאפשר להפעיל שוב

פתחו `http://localhost:4310/design-demo` והשוו בין שני התוצרים.

שאלו:

- מה התוצר הגנרי פספס לגבי ההחלטה שהמשתמש צריך לקבל?
- מה השתנה אחרי שהפעלתם UI/UX ו-design taste כ-Skills?
- אילו כללים מהעולם המקצועי שלכם שווה לארוז ב-Skill חוזר?

העתיקו את `workshop/templates/my-skill/` אל `.claude/skills/my-skill/`, מלאו את `SKILL.md` והריצו את ה-Skill על האפיון שלכם. שמרו את שתי ההרצות וההשוואה ב-`EVIDENCE/before-after.md`.

הראיה שלכם: `EVIDENCE/before-after.md` מציג לפחות שלוש החלטות שהשתפרו בזכות ה-Skill, ולא רק שינוי בסגנון.

## שלב 3 — ארכיטקטורה ותוכנית

הפעילו `/architecture-plan`. בקשו רשימת קבצים מדויקת, זרימת נתונים, מצבי כשל ובדיקות. עברו על התוכנית לפני שאתם נותנים אישור לביצוע.

אפשר להתקדם כשאתם יודעים להסביר למה זה החתך הקטן ביותר שעדיין נותן ערך מלא.

## שלב 4 — בנייה

ממשו את הפיצ׳ר לפי התוכנית. הפעילו מומחים רק על שאלות ממוקדות, והשאירו את ההחלטות והחיבור בין החלקים בסשן הראשי.

אל תמשיכו לפני שהפיצ׳ר עובד בנתיב אחד מקצה לקצה.

## שלב 5 — Loop Engineering

הפעילו `/loop-engineering`:

לפני הריצה, עברו על [`06-EXECUTION-LOOPS-HE.md`](06-EXECUTION-LOOPS-HE.md), נעלו Goal ובחרו את משטח ה-access הצר שמתאים למשימה.

1. בדיקה צרה.
2. `npm run verify`.
3. Review עצמאי באמצעות `change-reviewer`.
4. תיקון blockers בלבד.
5. עד שלושה סבבים.

עוצרים כשלכל קריטריון יש ראיה, או כשיש blocker ברור ומתועד.

## שלב 6 — הצגת העבודה בראיון

הכינו הסבר של שלוש דקות:

- איזו בעיה זיהיתם.
- איזו החלטה קיבלתם מעבר ל-ticket המקורי.
- מה האצלתם ולמי.
- איזו החלטה השארתם אצלכם.
- מה נכשל ואיך שיניתם בעקבותיו את הלולאה.
- אילו ראיות תומכות בתוצאה.
