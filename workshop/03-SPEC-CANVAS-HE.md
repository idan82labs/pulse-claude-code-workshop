# קנבס לאפיון פיצ׳ר

## 0. הכיוון של המשתתף

לפני שהמומחים מציעים פתרון:

- מה ראיתי כשפתחתי את המוצר?
- למי אני רוצה לעזור?
- איזו החלטה או פעולה אני רוצה לשפר?
- למה השינוי חשוב?
- מה לא ארצה לבנות עכשיו?

בקשו מה-Agent שלוש חלופות שמבוססות על ה-repo, בחרו או שנו אחת ושמרו את ההחלטה ב-`PRODUCT-DIRECTION.md`. המומחים מאתגרים את הכיוון; הם לא בוחרים אותו במקום המשתתף.

## 1. מה לא עובד היום

מה המשתמש לא מצליח להבין או לעשות? מה במוצר מוכיח שזו בעיה אמיתית?

## 2. המשתמש וההחלטה

מי המשתמש? איזו החלטה הוא מנסה לקבל, מתי ובאיזה מצב?

## 4. היעד

כתבו במשפט אחד מה צריך להשתנות בפועל, בלי לקבוע מראש איך הפתרון ייראה.

## 5. מה לא בונים עכשיו

מה נשאר בחוץ, גם אם הוא רעיון טוב?

## 6. הנחות ושאלות פתוחות

מה אנחנו יודעים, מה אנחנו רק מניחים, ואיזו תשובה עשויה לשנות את הפתרון?

## 7. Fan-out לזוויות מקצועיות

- מוצר: ערך, משתמש, החלטה ומדד.
- עיצוב: היררכיה, אינטראקציה, מצבים ונגישות.
- ארכיטקטורה: גבולות, נתונים, מצבי כשל ותחזוקה.
- QA: סיכונים, מצבי קצה וראיות.
- Security, רק כשיש trigger: הרשאות, מידע רגיש, input חיצוני, dependency, כלי, MCP, SSH או deploy.

כל מומחה מקבל את אותו brief, שאלה תחומה וחוזה פלט זהה. בסדנה מריצים ברצף כדי לשמור usage; במערכת מתאימה אפשר להריץ במקביל רק שאלות שאין ביניהן תלות.

## 8. הפתרון המסונתז

עשו fan-in: הסירו כפילויות, פתרו סתירות וחברו את ההמלצות להחלטה אחת ברורה. הציגו למשתתף החלטות שמשנות את הכיוון וקבלו את אישורו. ציינו מה התקבל, מה נדחה ומה השתנה — ולמה.

## אחרי אישור ה-Spec

רק עכשיו עוברים ל-Plan Mode:

```text
PRODUCT-DIRECTION.md → FEATURE_SPEC.md → /plan → PLAN.md → TASK LIST → IMPLEMENT
```

- ה-Spec מגדיר מה המוצר יעשה ולמה.
- ה-Plan מגדיר איך לבצע בקוד ובאיזה סדר.
- רשימת המשימות מנהלת את המסלול הנוכחי ומסמנת complete רק אחרי verification.
